# ADR-0001: Remove Mediator/CQRS Machinery, Adopt a Simple Layered Architecture

- **Status:** Accepted
- **Date:** 2026-07-03
- **Scope:** `lib/application` (application layer design for the whole project)

## Context

Phase 3 introduced an application layer patterned after MediatR-style CQRS:

- `IMediator` / `Mediator` — string-keyed request routing (`dispatch()` / `register()`)
- `ICommand` / `IQuery` with phantom result types, plus `ICommandHandler` / `IQueryHandler`
- A composable pipeline (`IPipelineBehavior`) with `ExceptionBehavior`, `LoggingBehavior`,
  and `ValidationBehavior` (with `IValidatableRequest`)

The implementation was correct, strongly typed, and passed all validation. The
question reviewed was not whether it worked, but whether it was necessary.

AlDaftar's actual requirements: a single organization, a single Next.js
application, a single Supabase database, a single deployment. No microservices,
no distributed messaging, no plugin architecture, no DI container.

An architecture review evaluated each abstraction against those requirements
and found:

- **The mediator solved a routing problem the project does not have.** In-process
  calls need no message routing; a direct import does the same job with
  compile-time checking and go-to-definition navigation.
- **It converted compile-time errors into runtime errors.** A missing import
  fails at build; an unregistered string `type` fails in production.
- **It required a composition root that is awkward in Next.js.** Server actions
  and route handlers are isolated serverless entry points; "register all
  handlers at startup" means a hand-maintained registration module re-executed
  per cold start. MediatR earns its keep in C# through container-based handler
  auto-discovery, which TypeScript-without-DI does not provide.
- **CQRS message classes only existed to feed the mediator.** With direct calls,
  a "command" is just a typed function parameter; read/write separation
  survives as method naming rather than type machinery.
- **The pipeline's guarantees were valuable, but its machinery was not.** Uniform
  logging, timing, and exception normalization matter for an accounting
  application; a middleware chain with ordering rules was an expensive way to
  get one function's worth of behavior.
- **Per-feature ceremony compounded.** Every future operation would have paid for
  a request class, a handler class, a string constant, and a registration —
  for the life of the project.

## Decision

Adopt a **simple layered architecture** with direct function calls between
layers, and reduce the application layer to a single abstraction.

1. **Deleted entirely:** `lib/application/contracts/`, `lib/application/mediator/`,
   `lib/application/pipeline/` — including `IMediator`, `Mediator`, `ICommand`,
   `IQuery`, `ICommandHandler`, `IQueryHandler`, `IValidatableRequest`, and all
   pipeline behaviors (313 → 69 lines, −78%).
2. **Kept and refactored:** `ApplicationService`
   (`lib/application/services/application-service.ts`) is now the only
   application-layer abstraction. Its `execute(operationName, callback)`
   method absorbed the pipeline's guarantees: execution timing, structured
   success/failure logging, exception normalization via `ErrorFactory.fromUnknown`
   (a thrown `AppError` passes through typed and intact), and wrapping every
   outcome in the `Result` pattern. `execute()` never throws.
3. **The layering:**
   - `lib/core` — Result pattern, typed immutable errors, structured logging,
     validation guards, technical types/utils (unchanged by this decision).
   - `lib/application` — concrete services extend `ApplicationService`, validate
     their input, do the work, and return `AsyncResult<T>`.
   - Entry points (later phases: server actions / route handlers) import
     services directly and `match()` on the Result at the boundary.

## Consequences

### Positive

- A new feature costs one service class and its input types — no message
  classes, no registration, no composition root.
- All routing is compile-time checked and IDE-navigable.
- The cross-cutting guarantees (timed structured logs, normalized errors,
  Result-only surfaces) are preserved in one place, `execute()`, and are
  simpler to test than a behavior chain.
- Less code to maintain: the layer shrank by 78% with no loss of capability
  actually required by this project.

### Negative / accepted trade-offs

- There is no single interception point through which _all_ operations flow;
  a new cross-cutting concern means extending `execute()` rather than adding
  a behavior. Accepted: one method on one base class is the cheaper edit.
- If the project ever genuinely needs message-based dispatch (background job
  queues, event sourcing, out-of-process handlers), this decision must be
  revisited. Accepted: none of those are on the roadmap, and reintroducing a
  dispatcher later is easier than carrying an unused one indefinitely.

## Alternatives considered

- **Keep the full mediator/CQRS stack** — rejected: pattern cost without the
  problems the pattern solves (see Context).
- **Keep the pipeline but drop the mediator** — rejected: behaviors have no
  host without a dispatcher; a higher-order function delivers the same
  guarantees without composition-ordering rules.
- **Framework-native only (no application layer at all)** — rejected: putting
  logging/error policy in every server action scatters exactly the code the
  review wanted centralized, and couples business operations to Next.js.
