| DL-033 | Audit persistence defect fixed (Review Board, 2026-07-05). **Proven:** a real form sign-in left the `Login` audit entry in the browser's `localStorage` (`aldaftar.audit`) instead of Supabase — the `audit` table stayed empty while the screen still showed entries. **Root cause:** `getAuditService()` is a module singleton; its first `record()` call happens on `/login`, which sits OUTSIDE the `(app)` layout where the persistence bootstrap runs — so the singleton captured the default LocalRecordStore and, across a client-side SPA session, kept routing every later entry to localStorage. **Fix (two parts, minimal):** (1) `AuditService` resolves its repository **per call** (getter over `getAuditRepository()`) instead of at construction, so it always reflects the registered provider; (2) `app/login/page.tsx` imports `components/app/persistence-bootstrap` so the Supabase provider is registered before the Login entry is written. **Re-verified live** on the production build via a real browser session: Login, Create, Update, Post, Logout, and Delete (attachment removal) all produce rows in the Supabase `audit` table with `localStorage.aldaftar.audit` empty; the Audit Log screen reads the same Supabase-backed service | A silently-empty audit trail is a data-integrity defect (BDD-010); fixed at the seam with no business-logic change | Active |
